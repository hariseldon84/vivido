## Fresh Strategic Ideas — No Legacy Bias

### Idea 1: "The Creator's Final Software"
Position as the LAST tool a creator ever needs to buy. Not a starting tool. Not a stepping stone. The destination. This flips the acquisition model — you market to creators who are fed up with the tool-switching treadmill.

### Idea 2: Film-Grade for Solo Creators
DaVinci Resolve is the closest thing to Vivido in power — but it was built for colorists and post-production studios. Vivido takes every DaVinci-grade capability (color science, audio rooms, VFX nodes) and makes them accessible without a film school degree. The UI language changes even if the underlying power doesn't.

### Idea 3: "Project Memory" as a Core Product Feature
Not a hidden feature. Not buried in settings. The compounding intelligence is THE marketing hook. "Vivido gets smarter the more you use it." Creators can see their style model building. This is the product's heartbeat.

### Idea 4: The Creator's OS — Not Just an App
What if Vivido isn't just an app, but acts as a creative layer on top of the OS? It can:
- Intercept screen/camera capture at the OS level (like OBS hooks into drivers)
- Manage the creator's entire asset library across projects
- Integrate with YouTube API natively (chapters, cards, thumbnails without leaving the app)
- Connect to the creator's channel analytics to inform editing decisions in real-time

### Idea 5: Multi-person Production Without Switching Tools
FCP has no real collaboration story. Premiere has Adobe Teams (enterprise). DaVinci has Studio (expensive). **No tool gives solo-to-team-to-studio collab at the creator price point.** Vivido can own this:
- Solo mode: Full power
- Duo mode: Two-person async editing (creator + editor relationship)
- Studio mode: Agency/team with roles, review flows, client approval

### Idea 6: Patent-Worthy Technology Areas
Per rawvision.md — filing patents is the goal. Potential patent areas:
1. **Per-creator style model** — ML model that accumulates editing preferences per channel
2. **Cross-project timeline intelligence** — surfacing patterns across 50+ projects
3. **YouTube-native primitive rendering** — Chapters, Cards, End Screens as first-class timeline objects
4. **Adaptive audio normalization per platform** — YouTube, Shorts, Podcasts each have different loudness standards; auto-normalize on export
5. **Cinematic AI pre-visualization** — AI generates rough cut from raw footage based on learned style, not generic rules

### Idea 7: Vivido as a Creator Movement (not just software)
rawvision.md talks about uniting creators. This is a community play:
- Vivido Certified Creator program
- Style sharing (opt-in) — learn from top creators' editing patterns
- Vivido Film Festivals — content made entirely in Vivido
- Open plugin ecosystem — community builds on top of Vivido's timeline

## Method 5: Kill the Competitors

The exact moment a creator uninstalls each competitor for Vivido:

**Killing Final Cut Pro**
Weakness: Mac-only, stateless, zero YouTube-native workflow.
Kill moment: Creator opens project 51. FCP knows nothing about projects 1–50. Same blank timeline. Same manual chapters in YouTube Studio. Same Canva thumbnail. FCP made them no faster on video 50 than video 1.
Vivido's weapon: Project 51 opens with a pre-staged rough cut, chapters suggested, Shorts marked — based on the style model from projects 1–50. You're reviewing, not building.
Windows bonus: FCP doesn't exist there. Vivido is the first FCP-grade experience on Windows.

**Killing Adobe Premiere**
Weakness: Subscription dependency, CC bloat, 5-tool post-export workflow, designed for broadcast TV in 1991.
Kill moment: Export an 8-minute YouTube video. Wait for CC to update. Export dialog with 47 settings. Then: YouTube Studio (chapters), separate Shorts upload, VidIQ session. 5 tools, 2 hours, 8-minute video.
Vivido's weapon: Chapters authored during editing. Shorts clipped during editing. One "Publish Package" generates everything. No subscription trap. Works offline.

**Killing DaVinci Resolve**
Weakness: Most powerful tool on Earth, steepest learning cliff, zero YouTube-native workflow, built for colorists not creators.
Kill moment: Creator opens DaVinci for color grading. Color Room has 47 panels. Fairlight looks like NASA. They spend 3 hours learning the interface without making one cut. Close laptop. Go back to Premiere, defeated.
Vivido's weapon: DaVinci-grade color science with creator vocabulary. "Make this cinematic" → 5 looks matching your channel's visual identity → one click. Same power, different language.

## Method 3: Minimum Legendary Product (MLP)

Not MVP. The smallest version of Vivido that already feels like a legend — where a creator finishes one real project and says "I didn't know editing software could feel like this."

**The four features that together constitute legendary (none alone is enough):**

**1. The Compounding Editor** — After each completed project, Vivido updates the creator's style model. By project 3 it pre-stages edits. By project 10, first-pass rough cuts that feel *like you made them*. Implementation: style model captures avg cut duration, hook length, B-roll ratio, audio treatment, chapter density. Rough cut uses these as weights — not generic excitement detection.

**2. YouTube-Native Timeline** — Chapters, Cards, End Screens, Shorts cut-points are draggable objects in the timeline. See chapter markers while editing. Shorts panel visible alongside long-form — mark clips as you edit. Export generates YouTube `.json` chapter file + standalone Shorts clips automatically. First time a creator drags a chapter marker in the timeline, they screenshot it.

**3. OS-Level Native Performance** — Full GPU/NPU/CPU, no cloud for basic ops. Real-time 4K scrubbing. No "rendering preview" spinners. Native Electron with FFmpeg native bindings + VideoToolbox (Mac) / NVENC-NVDEC (Windows). The first time a creator scrubs 4K at full speed, they feel it physically. That sensation IS the product.

**4. One-Click Professional Export** — Single button generates: Full YouTube MP4 (with chapter metadata) + Shorts clips (auto-cropped 9:16) + 3 thumbnail candidates + Platform-normalized audio. Eliminates 3–5 manual tool steps. First "Publish Package" click = the moment they tell 10 other creators.

**MLP intentionally excludes:** Color grading room, collaboration, recording capture, mobile, cloud render queue.

**MLP success criteria:** Creator's first Vivido project: time-to-publish-ready drops below 40% of current workflow. Zero required tool switches outside Vivido.

## Classification: Tree of Thoughts — What is Vivido, Actually?

Three parallel framings evaluated against: PRD clarity, build speed, 10-year ceiling, moat strength, GTM sharpness, founder's edge, rawvision alignment.

**Path A — Filmmaker's Desktop Tool:** Competes with FCP/DaVinci. Desktop-first, native GPU, local-first. Strong ceiling, but GTM is broad and build is slow.

**Path B — Creator Intelligence Platform:** Competes with Adobe Creative Suite. Per-creator style model is the core asset. Desktop is the interface, platform is the moat. Highest ceiling, best patent story, requires platform discipline from day one.

**Path C — YouTube Stack Killer:** Competes with the 7-tool creator workflow. Eliminates $1,500/yr of tools. Fastest GTM, sharpest positioning, time-boxed CapCut opportunity. Ceiling risks capping at "YouTube editor."

**Selected: B + C Hybrid**
- Start as Path C (YouTube stack killer — fastest to market, sharpest GTM)
- Architect as Path B (intelligence platform — style model + patents built from day one, invisible to users)
- Path A (filmmaker's tool) is the 2029 story — architecture must not prevent it

**Locked Classification:**
- Type: Desktop App (Electron, primary UX) + Web App (React+Vite, sync/collab) — platform-architected from day one
- Domain: Creator Economy → Filmmaker Tools (2026: YouTube entry, 2029: film expansion)
- Complexity: High
- Context: Greenfield
- One-line: *Vivido launches as the YouTube creator's final stack and grows into the filmmaker's OS.*

**Debate Club Showdown Verdict: Platform-first wins 4.5 — 0.5**

Synthesis (more useful than the score):
> Design as a platform. Ship as a desktop app.

**Recording Capability — Added to Vivido Scope**

Vivido records. Not just imports. Recording is a first-class feature — the session starts in Vivido and never leaves.

Recording modes:
- **Screencast** — Full screen, specific window, or custom region capture (replaces Loom, Riverside screen share)
- **Webcam** — Solo talking head, or picture-in-picture with screen (replaces Loom, Riverside)
- **Multi-track** — Screen + webcam + system audio + mic audio recorded as separate tracks simultaneously. Each track is a clean, uncompressed local file — not a compressed stream (Riverside's core differentiator, now inside Vivido)
- **Scene Composer** — Multiple sources with layout presets: Interview (two cameras), Tutorial (screen + corner cam), Podcast (audio-primary with visual), Cinematic (full webcam, no screen). Inspired by OBS scene system but with creator-friendly presets.
- **Local-first capture** — All recording writes to disk before any cloud operation. No recording loss from network drops. WAV for audio (uncompressed), source resolution for video — never compressed at capture (per CLAUDE.md architecture decisions)

Kill shots added:
- **Kills Loom** — Screen + webcam recording, instant project creation after capture
- **Kills Riverside.fm standalone** — Multi-track local recording, studio-quality guest recording (future: remote guest tracks via WebRTC, each recorded locally)
- **Kills OBS as standalone for non-streaming creators** — Scene composer covers tutorial/interview/podcast setups without OBS complexity

Recording → Timeline handoff:
- Capture ends → project auto-created → footage already in timeline → Whisper runs immediately
- No import step. No file management. Record stops, edit starts.

**Architecture Note (ADR-001-C extension):** Recording capture runs entirely in the Native Runtime (Electron main process). OS-level capture APIs (ScreenCaptureKit on Mac, Windows.Graphics.Capture on Windows) are invoked from the main process. Renderer shows the recording preview and controls via IPC. This keeps the UI responsive during capture and ensures local-first file writing at full quality.

  ADR-001-C (Proposed by: Kenji, Platform Architect)              
                                                                                                                                                            
  Title: Dual-Runtime with Explicit Compute Boundary              
                                                                                                                                                            
  Context: Same as above. This ADR rejects the false choice between A and B.                                                                                
                                                                                                                                                            
  Decision Proposed: Two distinct runtime environments with a shared data layer and explicit IPC contracts:                                                 
                                                                  
  Browser Runtime (Renderer Process): Owns all UI, interaction state, and lightweight media operations. WebCodecs for scrubbing preview                     
  (hardware-accelerated in browser). Waveform visualization via Web Audio API. Timeline state management. Transcript editing. Export configuration UI. All
  UI state lives here.                                                                                                                                      
                                                                  
  Native Runtime (Electron Main Process + Node.js): Owns all heavy compute. FFmpeg native for export, transcoding, audio extraction. Local Whisper inference
   (via whisper.cpp, C++ binding). Style model inference and update (lightweight JSON accumulation initially, ML later). File system operations. GPU compute
   for color grading operations that exceed browser capability.                                                                                             
                                                                  
  Shared Data Layer: Project file (open JSON format, documented). Asset manifest (references, not copies). Creator style model (local JSON, synced to       
  Supabase). All IPC between browser and native runtime goes through typed message contracts — no raw IPC.
                                                                                                                                                            
  The Contract: Browser runtime can always render. Native runtime handles operations that would block the UI. Result: UI never freezes waiting for FFmpeg.  
  Scrubbing is always smooth. Exports happen in the background via the native runtime.
                                                                                                                                                            
  Web App (Vercel): Shares 100% of the UI layer with the desktop renderer. Platform adapter detects context (Electron vs browser) and routes compute        
  requests accordingly. In browser: WebCodecs + WASM fallback. In Electron: native runtime via IPC.
                                                                                                                                                            
  Consequences — Positive:                                        
  - Best of both worlds: native performance for heavy compute, web portability for UI
  - UI never blocks — all heavy operations are async to the native runtime           
  - Web app and desktop share UI codebase — zero divergence               
  - Clear boundaries make testing tractable: browser runtime tests with standard tools, native runtime tests with Node.js test harness                      
  - Future: native runtime can run headless for cloud render queue (AWS Lambda or self-hosted) — same FFmpeg bindings, different host                       
                                                                                                                                                            
  Consequences — Negative:                                                                                                                                  
  - IPC contracts add latency for real-time operations that cross the boundary (e.g., frame-accurate preview during complex color operations)               
  - Two test environments, not one — browser tests AND native runtime tests                                                                                 
  - More complex initial architecture than A or B — requires discipline to maintain the boundary                                                            
                                                                                                                                                            
  Status: Proposed
